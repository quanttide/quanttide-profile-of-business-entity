# 全栈工程师笔试题


考核项目：量潮DevOps云
考核时长：2-3天，完成后以 Issue 或 PR 形式提交
提交物：一份工作蓝图，包含分析、设计和实现方案
工作环境：完全异步，模拟开源社区协作方式。过程中如有疑问，可通过 Issue 与我们讨论

## 一、背景说明

### 项目简介

DevOps 云是一个开源项目，定位是 DevOps 工具链之上的"第二大脑"。它不做代码托管、不做 CI 执行、不做制品存储，而是理解你用了哪些工具、这些工具之间应该怎么协作、并在需要时自动生成集成方案。

项目目前处于 0.3-0.4 版本，快速迭代中。开源地址：quanttide/qtcloud-devops。

### 核心概念

我们用两个概念来描述设计和开发方法：

· 判例：一个具体场景下的完整实现。比如"基于 GitHub + GitHub Actions + 官方 PyPI 的 CI/CD 配置"。它是可用的，但绑定具体平台。
· 宪法：从多个判例中提炼的抽象规则。比如"代码仓库"这个概念本身——不管它是 GitHub、GitLink 还是 GitLab，它们都有共同的属性和行为。宪法定义共同点，不绑定任何厂商。

我们的开发过程，就是不断把判例中的共性提取为宪法，然后让新场景自动复用宪法的过程。

### 当前状态

平台目前主要适配了 GitHub，正在接入第二个代码托管平台 GitLink。代码中还存在一些硬编码的特定平台假设，需要被识别并拆解为可扩展的抽象模型。

## 二、笔试任务

### 任务一：硬编码假设分析

我们的代码中，有一些对特定平台（如 GitHub）的假设写死在代码里。这些假设会成为接入新平台的阻碍。

请你：

1. 拉取项目代码，找出你认为存在硬编码平台假设的 3-5 个关键位置。
2. 对每一处，说明：当前假设了什么、如果要支持 GitLink 会遇到什么问题。
3. 提出修改思路：你认为应该怎么改，才能让这部分代码不依赖任何具体平台。

### 任务二：设计"代码仓库宪法模型"

假设我们要同时支持 GitHub、GitLink、GitLab、以及用户自建的内部 Git 服务。它们各有不同的 API、认证方式、特性。

请你：

1. 设计一个通用的"代码仓库"抽象模型。定义出你认为最核心的属性和方法。
2. 解释你的设计决策：为什么选择这些属性和方法？你舍弃了什么？为什么可以舍弃？
3. 说明这个模型如何处理以下差异：
   · GitHub 用 OAuth，自建 Git 服务用 SSH Key
   · GitLink 的 API 返回格式与 GitHub 不同
   · 自建服务没有 Webhook 功能

### 任务三：设计"新平台接入向导"

假设宪法模型已经实现，并且已经有 GitHub 和 GitLink 两个判例。现在我们希望实现一个"新平台接入向导"——用户只需要输入一个未知类型的代码仓库地址，系统尽可能自动完成集成。

请你：

1. 设计这个向导的技术方案，包括关键组件、流程步骤、技术选型。
2. 画出架构图（可以用 ASCII、草图照片、或在线绘图工具）。
3. 说明：当自动适配失败时，系统如何优雅降级，如何引导人类介入？

## 三、提交要求

提交形式：将你的工作蓝图以 Markdown 文档提交到仓库 quanttide/quanttide-roadmap-of-devops。可以是一个 Pull Request，也可以是一个 Issue。

内容结构：

· 对三个任务的分析和设计
· 必要的图表（架构图、流程图）
· 你认为我们现有代码或设计中存在的问题和改进建议（如有）
· 你在分析和设计过程中产生的问题（哪些地方不确定？你做了哪些假设？）

评估维度：

· 对项目核心概念（宪法/判例）的理解深度
· 抽象建模能力：能否从具体中提炼普遍规则
· 工程判断力：设计决策是否有清晰的取舍理由
· 沟通质量：文档是否清晰、结构是否易读、假设是否显式说明

## 四、关于工作方式

我们的协作模式类似开源社区：

· 异步沟通，文档优先
· 设计方案先于代码实现
· 每一个设计决策都需要说明"为什么"
· 鼓励在 Issue 区提问和讨论

期待看到你的工作蓝图。有任何疑问，欢迎在 Issue 中直接提出。
