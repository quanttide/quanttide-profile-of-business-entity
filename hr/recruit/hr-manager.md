# 人事经理笔试题

考核项目：量潮 · 人力资源
考核时长：2-3天，完成后提交
提交物：一份工作方案
工作环境：远程异步，模拟实际工作节奏

—

一、背景说明

我们怎么做招聘

我们的招聘不考算法题，不考八股文。

我们把日常教学的方法论搬进了招聘流程。全栈、算法、数据三个方向的笔试题，都是基于开源项目的真实问题。候选人 2-3 天完成一份工作蓝图，我们根据他的分析过程和设计思路来评估。

这套方法已经跑通。它筛选出的人，入职后能直接进入工作节奏。

我们怎么带新人

我们内部验证过一件事：十年老手的工作体系，可以在一周内成套传递给新人。

一个大一文科生，学了七天 Vibe Coding，开始独立设计产品，甚至推翻了十年老手的技术方案。

我们管这套方法叫"开全图"——把复杂系统的全景图展示出来，用精良的示范和精准的解释，让新人在实战中快速抓住核心意图。

你需要做什么

我们需要一个能理解这套方法论、并能把它落地到招聘和新人培养流程中的 HR。

参与招聘考核的设计，确保它持续筛选出和我们思维结构同频的人。你还需要把"一周新人"的成功经验，转化为可复用的新人培养流程。

—

二、笔试任务

任务一：理解我们的方法论

请阅读以下材料（我们会提供）：

· 《重构是 Vibe Coding 时代的必备技能》
· 《如何快速向新人成套传递十年老手的工作体系？》
· 全栈工程师笔试题
· 算法工程师笔试题
· 数据工程师笔试题

回答：

1. 用你自己的语言，解释量潮的核心方法论是什么。不要复述原文，用你理解后的方式表达。

2. 这套方法论在招聘笔试中是怎么体现的？请以全栈工程师笔试题为例，说明这套题在考什么、为什么这样考、它能筛选出什么样的人。

3. "一周新人推翻十年老手"这个案例中，哪些关键动作促成了新人的快速成长？这些动作是否可以标准化为一套新人培养流程？请给出你的初步设想。

—

任务二：设计一个岗位的考核方案

假设我们需要招聘一个目前还没有笔试方案的岗位（比如：产品经理、UI/UX 设计师、或者你认为量潮当前需要的任何岗位）。

请你：

1. 确定这个岗位的核心能力要求。在我们这套方法论下，这个岗位最需要的是什么？为什么是这些能力，而不是其他？

2. 设计一套笔试题。参考我们已有的笔试题结构，为这个岗位设计一套 2-3 天完成的笔试方案。需要包含：

· 背景说明（让候选人理解我们在做什么）
· 笔试任务（2-3 个具体任务，层层递进）
· 提交要求和评估维度

3. 说明你的设计逻辑。这套题为什么这样设计？每道题在考什么？你想通过这套题筛选出什么样的人？

—

任务三：设计新人培养流程

基于我们"开全图"的教学理念和"一周新人"的成功案例，请你设计一套新人入职后的培养流程。

要求：

1. 流程的时间跨度和阶段划分。第一个月，新人需要经历哪几个阶段？每个阶段的目标是什么？

2. 每个阶段的关键动作。新人做什么？导师做什么？需要什么资源或文档支持？

3. 考核和反馈机制。如何判断新人是否通过了某个阶段？谁来判断？标准是什么？

4. 流程的迭代机制。这套培养流程本身如何持续优化？如果某个新人成长速度远超或远低于预期，流程怎么适应？

—

三、提交要求

提交形式：一份 Markdown 文档。

内容结构：

· 任务一：方法论理解 + 笔试题分析 + 新人成长关键动作提炼
· 任务二：选定岗位 + 能力要求 + 完整笔试题方案 + 设计逻辑说明
· 任务三：新人培养流程设计（阶段、动作、考核、迭代）
· 你的问题和假设

评估维度：

· 方法论理解深度：是否真正理解了"宪法与判例"、"开全图"、"重构即学习"这些核心概念，而不是停留在名词解释
· 迁移应用能力：能否将已有方法论迁移到新岗位的考核设计中，而不是简单照搬
· 流程设计能力：新人培养流程是否具体、可执行、有反馈闭环
· 用户视角：设计考核和培养流程时，是否始终在想"候选人/新人会怎么体验这个过程"
· 文字表达质量：文档是否结构清晰、逻辑自洽、能用简洁的语言说明复杂的事情

—

四、关于工作方式

· 异步协作，文档优先
· 每个决策需要说明"为什么"
· 鼓励在过程中提问

—
